## 550 天全流程模拟与报告

**任务**：搭建 headless 模拟环境，模拟正常玩家 550 天游戏流程，覆盖各系统机制，查找漏洞与数值问题。

**模拟器搭建**（`simulation/headless/`）：
- 用 esbuild 把 `src/core` 打包为 CJS bundle，Node 中 monkey-patch rAF 手动推进天数
- `sim-core.cjs`：Game/状态/事件/命令执行/数据采集封装；修复 EventBus 通配符 bug（monkey-patch emit）
- `player.cjs`：玩家策略驱动器，封装 31 种命令调用；修复 startExperiment computeRatio 默认值、installAllCards 本地槽位计数
- `main-sim.cjs`：550 天四阶段策略编排（survive/expand/scale/frontier）

**模拟结果**：
- 最终：资金 $920.6M，员工 31，模型 3 个（7B/13B/30B），技术 23 项，卡 574/704
- 覆盖：31 种命令、60 种事件、15/17 系统
- 70B/100B 未训练成功（once+rejected 模拟器 bug + 技术/硬件门槛）

**发现的程序漏洞（6 项）**：
1. RaiseFundingCommand 对非法 FundingType 无校验 → 资金 NaN
2. StartExperimentCommand 对 undefined computeRatio 无校验 → 实验卡死
3. 后训练无 StartPostTrainingCommand（架构缺口）
4. EnterRegionCommand 对未知 regionId 静默 return
5. DC 地点 id（nv_us/or_us/iceland/singapore）与地区 id（us-west 等）体系不一致
6. inverse 能力维度（hallucination_rate/sycophancy/eval_awareness）在 capabilityCalc/marketCalc/RiskSystem 中未反转

**发现的数值问题（5 项）**：
1. 训练崩溃率偏高（0.001×parallelSize×100/infraReliability，上限 0.1/天）
2. 实验成熟度提升慢（598 次实验后 tensor_parallel 仅 94%）
3. 节点互联带宽门槛陡峭（PCIe5 128 → NVLink3 600 跨度 4.7 倍）
4. 大模型训练可行性窗口过窄（70B 需 TP+PP+高带宽同时满足）
5. 后期硬件故障率高（36 卡 + 28 节点）且无维护机制

**输出**：`simulation/headless/SIMULATION_REPORT.md`（含操作流水、数据曲线、系统覆盖清单、漏洞/数值问题及修复建议）

## 大规模卡（数十万级）性能优化

**任务**：排查并修复数十万卡后的卡顿，检查所有显卡数量/基建相关代码的性能瓶颈。

**基准测试发现**（修改 5 卡/天）：20 万卡每日 produce 开销 33ms → 优化后 3.5ms。

**修复 3 个瓶颈**：
1. **主因** `GameState.ts`：`setAutoFreeze(false)`。immer finalize 深度冻结遍历整个卡池，与卡数线性正相关，是每日卡顿最大来源（~10× 提升）。结构共享不受影响，WeakMap 缓存失效机制照常。
2. `cardIndex.ts`/`infraIndex.ts`/`computeUtilization.ts`：用 immer `isDraft()`+`original()` 把 draft 解析回 base state 再查 WeakMap 缓存。修复 draft proxy 引用≠base 引用导致每日每系统全量重建索引。注意：初版误用 `__DRAFT_STATE__` 字符串属性（immer 内部用 Symbol），改为官方 `original()`。
3. `InfraCommands.ts` InstallCardCommand：新增 `getNodeInstalledPowerKW` WeakMap 缓存（按 installedCards 引用），批量装 N 卡功耗检查 O(N²)→O(N)。

**确认无需改动**：summarizeCardSpecsFromMap 型号聚合、故障采样、 suspendNotify 合并渲染、UI 卡数>500 折叠聚合行等已有优化良好。

**输出**：`docs/PERFORMANCE.md`（瓶颈清单 + 残余 P2/P3 风险：offline 卡反查表、热备短路、tflops 增量缓存）。

## 修复 5 个残余性能风险（第二轮）

1. **P2 offline 反查表**：`cardIndex.ts` 新增 `getOfflineCardIndex()`（按 base 引用 WeakMap 缓存待恢复 offline 卡）。`InfrastructureFailureSystem` 恢复循环从遍历所有已装卡 O(已装卡数) 降为 O(offline 数)，数十万卡下每日从数十万次查询降为个位数。
2. **P2 热备短路**：`cardIndex.ts` 新增 `getBrokenCardIndex()`。`applyHostSpares` 入口 `brokenIdx.size===0` 整体短路；有 broken 卡时遍历反查集合 + `dcNodeIds` 归属判定，不再扫描健康卡。
3. **P3 tflops 缓存**：`calcClusterTotalTflops` 单条变量 → `WeakMap<base引用, number>`，日内多次 resourceMeta 变化只在新引用首次重建。
4. **P3 resourceMeta 订阅**：经分析现有 getCardIndex WeakMap 缓存 + suspendNotify 合并渲染已足够，无需拆分。
5. **P3 diagnoseTraining**：重构为按型号聚合（byModel Map），消除大数组、Math.min(...大数组) RangeError、serverNodes.find O(N²)，改用 getNodeIndex O(1)。

**验证**：`tsc -b` 通过；immer 基准确认反查索引正确（offline=2/broken=1）、draft 缓存复用 20 次仅 0.7ms。
